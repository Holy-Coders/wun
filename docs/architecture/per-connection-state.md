# Per-connection state, durability, and cross-instance scaling

## What changed

Up to phase 1.D, every Wun server held a single global `app-state`
atom and intent morphs ran against it. That model was deliberately
simple for the transport-and-diff spike, and it meant "log in" was
shared across every connected client -- correct for a single-tenant
demo, broken for anything multi-user.

Phase 1.E flips the default. State now lives per-connection in
`wun.server.state/conn-states`, a map keyed by the server-issued
`conn-id`. Each SSE connection has its own slice. Intent morphs run
against the originating slice; broadcasts go only to that connection.
There is no fan-out at the framework level -- cross-conn updates are
an app-level concern (see "Cross-conn writes" below).

## State at three layers

```
                  ┌─────────────────────────────────┐
                  │ in-memory:  conn-states atom    │
                  │   {conn-id -> state-map}        │
                  └────────────┬────────────────────┘
                               │ persist watcher (per-app)
                               ▼
                  ┌─────────────────────────────────┐
                  │ durable:    wun_conn_state       │
                  │             table / Datomic ent  │
                  └────────────┬────────────────────┘
                               │ pg_notify (postgres only)
                               ▼
                  ┌─────────────────────────────────┐
                  │ cross-instance:                  │
                  │   LISTEN wun_state_change        │
                  │   ↳ reload + re-broadcast        │
                  └─────────────────────────────────┘
```

The framework owns the in-memory layer. The other two are opt-in
fragments that ship with `wun new app foo --db <x>`.

### Layer 1: in-memory (framework)

`wun.server.state/conn-states` is a single atom holding every live
slice. The framework exposes:

```clojure
(state/state-for conn-id)                ;; read
(state/swap-state-for! conn-id f & args) ;; write (atomic)
(state/register-init-state-fn! f)         ;; what fresh slices look like
(state/register-state-watch! f)           ;; called on every change
(state/drop-state-for! conn-id)           ;; explicit eviction
```

The intent handler in `wun.server.http` calls `swap-state-for!` with
the conn-id from the incoming envelope. The render path calls
`state-for`. That's it -- no other code reads or writes state
directly.

### Layer 2: durable (per-app, opt-in)

Generated by `wun new app foo --db <sqlite|postgres|datomic>`. The
fragment ships:

- A `wun_conn_state` table (or Datomic schema attrs).
- A `myapp.persist` namespace that registers a state-watch.

Whenever a slice has `:session/user-id` set, the watch upserts the
session-stripped slice to the table keyed by `user_id`. Anonymous
slices are NOT persisted -- the web client's localStorage hot-cache
already handles reload-survival for that case, and persisting
anonymous slices keyed by ephemeral conn-id would just churn rows.

What lives in a slice:

- `:session` (token + user-id + email; regenerated on each login,
  not persisted)
- `:notes` (the demo CRUD list; re-fetched from its canonical table
  on every load -- not persisted as part of the slice)
- everything else (counters, drafts, UI prefs)

`myapp.persist/excluded-keys` is the one place to change that
shape.

### Layer 3: cross-instance (Postgres only)

Generated by `--db postgres`. The fragment ships
`myapp.server.bus`, which:

1. After every successful persist write, calls
   `pg_notify(wun_state_change, '<user-id>')`.
2. Holds a dedicated background thread per process that runs
   `LISTEN wun_state_change` and polls `getNotifications()` every
   second.
3. On notification, finds every conn-id in *this* instance's
   `conn-states` whose slice has the matching user-id, reloads the
   slice from `wun_conn_state`, and re-broadcasts.

The dedicated connection lives outside HikariCP because the pool
multiplexes connections across requests and would lose `LISTEN`
state. Reconnects on errors with five-second backoff.

SQLite and Datomic are single-instance by construction in this
scaffold -- no cross-instance bus. Run them with one replica.

## Walkthrough: log in on instance A, see your data on instance B

```
[browser1 -> instance A]                [browser2 -> instance B]
   POST /intent log-in
     A: swap-state-for! conn-A
        sets :session in conn-A's slice
     A: persist watch fires
        UPSERT wun_conn_state user=42
     A: pg_notify wun_state_change '42'
                                            B: receives notification
                                            B: reload state_edn for 42
                                            B: any conn whose slice has
                                               user-id=42? merge in saved
                                               state, broadcast SSE patch
   <- SSE patch (state delta)
                                            <- SSE patch (state delta)
```

A user logged in on browser1 (served by instance A) and browser2
(served by instance B) sees the same `:counter` value within ~one
poll-tick of latency.

## Cross-conn writes (chat rooms, leaderboards, etc.)

Genuinely shared state is the unusual case, and the framework
deliberately has no `broadcast-to-all-conns` affordance for it.
When you do need it, the recipe is:

```clojure
(definent :myapp/post-message
  {:params [:map [:body :string]]
   :morph
   (fn [state {:keys [body]}]
     ;; Append to your local view, then broadcast to everyone.
     (let [msg {:from (get-in state [:session :email])
                :body body
                :at   (java.util.Date.)}]
       #?(:clj
          (do (chat-store/save! msg)
              (doseq [other-cid (keys @wun-state/conn-states)
                      :when (not= other-cid (current-conn-id))]
                (wun-state/swap-state-for! other-cid update :messages
                                            (fnil conj []) msg))
              (assoc state :messages (chat-store/recent 50)))
          :cljs
          (update state :messages (fnil conj []) msg))))})
```

That walks `conn-states` itself and fans out. Each touched slice
fires its own watch, persists, and (on Postgres) emits its own
NOTIFY. Other instances pick the change up via the bus.

## What's still missing

- **Auto-rehydrate on SSE reconnect.** The handshake doesn't yet
  carry a session token, so a fresh conn-id can't be tied back to a
  saved user-id at connect time. Once it does, the init-state-fn
  will call `(persist/load-state-by-user user-id)` and fold the
  result into the new slice. Today this happens via the login
  intent (which sets :session, triggers the watch, etc.) on the
  next interaction.
- **Slice eviction policy.** Closed connections leave their slices
  behind. The `gc-tick!` in http.clj prunes the SSE channel map
  but not `conn-states`. Real-world apps want a TTL or an eviction
  on `state/remove-connection!`. This is intentionally not wired
  up yet because durability without auto-rehydrate would lose the
  user's state on reconnect.
- **Datomic cross-instance bus.** Datomic Local is in-process by
  design. For a multi-instance Datomic deployment you'd switch to
  Datomic Cloud (or Pro) and use its own change feed, OR keep the
  Postgres LISTEN/NOTIFY pattern by adding a separate Postgres
  side-channel just for notifications.

## Migrating an existing app

If you have a running app on the previous global-state model, the
break is mechanical:

1. `@wun.server.state/app-state` → `(wun.server.state/state-for conn-id)`
2. `(swap! wun.server.state/app-state f args)` →
   `(wun.server.state/swap-state-for! conn-id f args)`

The `migrations/0001_per-conn-state.bb` codemod handles the
mechanical rewrite for `(swap! state/app-state ...)` patterns and
flags reads that need a `conn-id` in scope.

Behaviour change to be aware of: morphs that accumulated shared
state (a global counter, "online users") will now be per-user. To
preserve the old behaviour for one specific intent, walk
`@conn-states` from the morph and apply the delta to every slice.
That's almost never what you want -- usually the per-user shape
turns out to be the actual feature.
